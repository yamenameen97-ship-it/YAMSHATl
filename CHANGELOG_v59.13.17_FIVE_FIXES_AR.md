### FIX #1 — زر "إبلاغ عن رسالة" في الشات يُرسل فعلياً ✅

**المشكلة (خطيرة):**
في `pages/Chat.jsx` (السطور 2035-2040 سابقاً)، عند الضغط على "إبلاغ" لرسالة في الشات الفردي،
كان الكود:
```jsx
onReport={(message) => {
  const reason = window.prompt('سبب الإبلاغ عن الرسالة:', '');
  if (reason !== null && reason.trim()) {
    pushToast({ type: 'success', title: 'تم إرسال البلاغ', description: 'سيراجعه فريق الإدارة قريباً' });
  }
}}
```
**أي:** كان يُظهر Toast كاذب يدّعي أنّ البلاغ أُرسل، **لكنه لم يستدعِ أي API إطلاقاً** —
البلاغ لم يكن يصل لـ admin panel ولا يُسجَّل في قاعدة البيانات. مماثل تماماً لـ FIX #5 الذي
أُصلح في `ProFeedPostCard` في v59.13.16، لكنه ظلّ موجوداً في الشات.

بالإضافة لذلك: `window.prompt` يجمّد الواجهة ومحجوب في بعض المتصفحات.

**الحل:**
- استيراد `ReportModal` في `pages/Chat.jsx`.
- إضافة state `reportTarget`.
- استبدال `window.prompt` + Toast كاذب بـ `setReportTarget({ id, label })`.
- ربط `<ReportModal targetType="message" />` في نهاية الـ JSX.
- البلاغ الآن يُرسَل عبر `POST /api/reports` بنفس آلية البلاغات الموحّدة، ويظهر في لوحة الإدارة.

### FIX #2 — تعديل الرسالة في الشات عبر مودال داخلي بدل window.prompt ✅

**المشكلة:**
في نفس الملف (`pages/Chat.jsx` سطر 2014)، تعديل الرسالة كان:
```jsx
const newText = window.prompt('تعديل الرسالة:', message?.content || '');
```
- يجمّد الواجهة بالكامل (واجهة الشات + لوحة المفاتيح على الموبايل).
- لا يدعم نصوصاً متعددة الأسطر.
- محجوب في بعض المتصفحات و في PWA المثبّت.
- لا توجد عدّاد للأحرف ولا مفتاح ESC ولا تأكيد مرئي.

**الحل:**
- إضافة state `editingMessage` + `editingDraft`.
- مودال داخلي بـ `<textarea>` متعدد الأسطر (4 أسطر، قابل للتغيير).
- زرّان واضحان: "إلغاء" و"حفظ التعديل" + زر الحفظ مُعطَّل إذا لم يتغيّر النص.
- عدّاد الأحرف (حد 2000) + إغلاق بـ ESC + إغلاق بالنقر على الخلفية.
- تطبيق `applyMessagePatch` فقط بعد ضغط الزر صراحةً.

### FIX #3 — توحيد الإبلاغ في رسائل المجموعة + مودال إعادة توجيه ✅

**المشاكل في `pages/GroupChat.jsx`:**

1. **زر "إبلاغ"** (سطر 605 سابقاً) كان يستخدم `window.prompt('سبب البلاغ:')` —
   تناقض مع نظام ReportModal الموحّد المستخدم في باقي التطبيق (المنشور، الريل، الستوري).

2. **زر "إعادة التوجيه"** (سطر 595 سابقاً) كان يستخدم `window.prompt` لإدخال اسم المستخدم —
   لا يوجد placeholder، لا يدعم اللصق المريح، يجمّد الواجهة.

**الحل:**
- استيراد `ReportModal`.
- استبدال `window.prompt` للإبلاغ بـ `setReportTarget` يفتح `<ReportModal targetType="group_message" />`.
  - **ملاحظة معماريّة:** ظلّ استدعاء `reportGroupMessage` المُخصَّص للمجموعات متاحاً، لكن الواجهة الرئيسية الآن
    موحّدة عبر ReportModal الذي يكتب إلى endpoint `/reports` الموحّد ويظهر في `AdminReportsPanel`.
- استبدال `window.prompt` لإعادة التوجيه بمودال داخلي:
  - حقل إدخال نصّي مع placeholder واضح (`@username` أو `group:id`).
  - Enter للإرسال، ESC للإلغاء.
  - زر "إعادة توجيه" مُعطَّل إذا الحقل فارغ + حالة busy أثناء الإرسال.
  - منع الإغلاق العَرَضي بالنقر على الخلفية إذا كان الإرسال قيد التنفيذ.

### FIX #4 — حوار تأكيد داخلي لإزالة صديق في FriendsAll.jsx ✅

**المشكلة (تناقض داخلي بالنظام):**
في v59.13.15 تم إصلاح هذه المشكلة في `pages/Friends.jsx` (تم استبدال `window.confirm` بحوار مخصص)،
**لكن `pages/FriendsAll.jsx` (صفحة "عرض الكل") ظلّ يستخدم:**
```jsx
if (type === 'unfriend' && !window.confirm('هل تريد إزالة هذا الصديق؟')) return;
```
أي أنّ المستخدم يحصل على تجربة متناقضة بين الصفحتين، والمشاكل الموثَّقة في v59.13.15 (حجب المتصفّحات، تجربة موبايل سيئة) ما زالت قائمة هنا.

**الحل:**
- إضافة state `confirmUnfriend` (يُخزِّن `friendshipId` المراد إزالته).
- عند ضغط "إزالة" → نفتح مودال داخلي بـ "هل تريد إزالة هذا الصديق؟" + شرح "يمكنك إعادة إرسال طلب صداقة لاحقاً".
- بعد التأكيد → نُعيد استدعاء `handleAction` بنوع `'unfriend-confirmed'` الذي يتعامل معه نفس فرع `removeFriendship` (مع تحويل `busyKey` إلى `'unfriend'` للحفاظ على تطابق الـ UI).
- ESC + نقر على الخلفية + autoFocus على زر "نعم" + RTL + Noto Sans Arabic.

### FIX #5 — مودال إضافة نص للقصة بدلاً عن window.prompt ✅

**المشكلة:**
في `components/stories/StoryEditor.jsx` سطر 67، عند ضغط زر "إضافة نص" أثناء تحرير قصة:
```jsx
const text = window.prompt('أدخل النص:');
```
- على الموبايل: يفتح حوار النظام البدائي الذي يكسر سياق التحرير ويُخفي معاينة القصة.
- لا توجد طريقة للصق نصوص متعددة الأسطر بسهولة.
- محجوب في PWA المثبّت كتطبيق.
- لا يوجد حد للأحرف (يمكن لصق روايات كاملة).
- لا يدعم RTL أو خط القصة.

**الحل:**
- إضافة state `showTextModal` + `textDraft`.
- مودال متراكب فوق محرر القصة (z-index: 10000) يحفظ السياق البصري للمستخدم.
- `<textarea>` 3 أسطر، حد 140 حرف، RTL، خط Noto Sans Arabic.
- اختصارات: `ESC` للإلغاء، `Ctrl/Cmd+Enter` للإضافة (ظاهرة كتلميح).
- زرّان: "إلغاء" و"إضافة" + الزر مُعطَّل إذا الحقل فارغ.
- عدّاد أحرف مرئي (140/140).

# Yamshat — دليل الربط والتشغيل على Render

## 1) شكل الربط الصحيح
المشروع بعد التعديل شغله الموصى به هو **خدمة ويب واحدة** على Render:
- الـ Backend Flask يخدم الـ API على `/api`
- نفس الخدمة تخدم ملفات الـ Frontend من مجلد `frontend`
- بالتالي التطبيق لا يحتاج ضبط عنوانين مختلفين للواجهة والخلفية داخل المتصفح طالما الفتح من نفس الدومين

## 2) أهم سبب لعدم حفظ البيانات سابقاً
السبب الأكبر كان وجود **ربط قاعدة بيانات غير موحّد** داخل المشروع:
- معظم الـ API الأساسية كانت تستخدم `backend/db.py` وتعتمد على `DATABASE_URL`
- بعض أجزاء قديمة كانت تعتمد طبقة أخرى/أسلوب قديم
- تم توحيد جزء طلبات الصداقة مع نفس قاعدة البيانات الأساسية حتى تبقى العمليات محفوظة في نفس PostgreSQL

لو `DATABASE_URL` غير موجود أو كان يشير لقاعدة مختلفة، هتظهر أعراض مثل:
- الحساب يدخل لكن بعض البيانات لا تظهر
- طلبات الصداقة أو بعض العمليات تتحفظ في مكان مختلف
- اختلاف بين ما يظهر في الواجهة وما هو موجود فعلياً

## 3) متغيرات البيئة المطلوبة فعلاً
### متغيرات أساسية إجبارية
- `DATABASE_URL` : رابط PostgreSQL الكامل من Render
- `SECRET_KEY` : مفتاح جلسات Flask
- `JWT_SECRET_KEY` : مفتاح JWT

### متغيرات ربط الدومين والجلسة
- `FRONTEND_ORIGIN` : رابط الواجهة النهائي
- `BACKEND_ORIGIN` : رابط الخدمة النهائي
- `RENDER_EXTERNAL_URL` : نفس رابط الخدمة العامة على Render
- `ALLOWED_ORIGINS` : قائمة origins مفصولة بفواصل
- `SESSION_COOKIE_SECURE=1`
- `SESSION_COOKIE_SAMESITE=None`
- `SESSION_DAYS=30`

### متغيرات الإدارة
- `ADMIN_EMAILS` : قائمة بريد الأدمن مفصولة بفواصل
- `ADMIN_USERNAMES` : قائمة أسماء الأدمن مفصولة بفواصل

### متغيرات البث المباشر LiveKit
- `LIVEKIT_WS_URL` : رابط WebSocket الخاص بـ LiveKit ويبدأ غالباً بـ `wss://`
- `LIVEKIT_URL` : بديل/نسخة احتياطية لو محتاج نفس القيمة
- `LIVEKIT_API_KEY`
- `LIVEKIT_SECRET`

### متغيرات الإشعارات Firebase
واحد فقط من الاثنين يكفي:
- `FIREBASE_SERVICE_ACCOUNT_JSON`
- `FIREBASE_SERVICE_ACCOUNT_PATH`

### متغيرات اختيارية لكن مفيدة
- `REDIS_URL`
- `CELERY_BROKER_URL`
- `CELERY_RESULT_BACKEND`
- `DEFAULT_COIN_BALANCE`
- `LOG_LEVEL`
- `MAX_CONTENT_LENGTH_MB`

## 4) هل في Frontend keys مطلوبة؟
في الشكل الحالي **لا توجد مفاتيح سرية لازم تتحط داخل الـ Frontend** لتشغيل الموقع العادي.

الواجهة تعتمد على:
- `/api/app-config.js`
- نفس origin الذي تم فتح الموقع منه
- التخزين المحلي لحفظ الجلسة والستايل والثيم فقط

يعني:
- مفاتيح LiveKit السرية **لا** توضع في الواجهة
- مفاتيح قاعدة البيانات **لا** توضع في الواجهة
- مفاتيح Firebase service account **لا** توضع في الواجهة

لو التطبيق الأندرويد يستخدم FCM من جهة العميل، يبقى فقط ملف إعداد العميل مثل `google-services.json` في تطبيق Android، لكن ليس داخل صفحات الويب.

## 5) القيم الموصى بها على Render
لو الخدمة المنشورة هي:
- `https://yamshat-1.onrender.com`

استخدم تقريباً:
- `FRONTEND_ORIGIN=https://yamshat-1.onrender.com`
- `BACKEND_ORIGIN=https://yamshat-1.onrender.com`
- `RENDER_EXTERNAL_URL=https://yamshat-1.onrender.com`
- `ALLOWED_ORIGINS=https://yamshat-1.onrender.com,http://127.0.0.1:5500,http://localhost:5500,http://127.0.0.1:5000,http://localhost:5000,capacitor://localhost,ionic://localhost`

## 6) ربط البث المباشر
البث لن يعمل إلا إذا كانت هذه الأربعة موجودة وصحيحة:
- `LIVEKIT_WS_URL`
- `LIVEKIT_API_KEY`
- `LIVEKIT_SECRET`
- خدمة LiveKit نفسها شغالة وقابلة للوصول

بدونهم، الواجهة ستفتح لكن إنشاء الغرفة أو الانضمام سيعطي فشل إعداد أو عدم اتصال.

## 7) ما الذي تم تعديله داخل الكود
- حفظ دور المستخدم `role` داخل الجلسة الأمامية لإظهار صلاحيات الأدمن بشكل صحيح
- صفحة `admin.html` أصبحت تعمل من نفس جلسة الموقع بدل تحويل خارجي يفقد الجلسة
- تحسين روابط الريلز بحيث تبقى صالحة بعد النشر والخروج والعودة
- إصلاح عرض الستوري بالاعتماد على رابط وسائط مضبوط
- جعل زر المتابعة يتحدث فوراً إلى `إلغاء المتابعة`
- إضافة زر `طلب صداقة` مع حالات: إرسال / إلغاء / قبول
- تقليل مشاكل الكتابة في التعليقات عبر منع التحديث التلقائي من قطع الإدخال أثناء الكتابة
- إضافة 3 أنماط واجهة: الكلاسيكي، إنستغرام، والثالث Neon
- تقوية إنشاء المجموعات عبر فهارس تمنع التكرار في الأعضاء
- توحيد مسار حفظ طلبات الصداقة مع نفس قاعدة PostgreSQL الأساسية

## 8) خطوات النشر بعد رفع الملف
1. ارفع الملفات المعدلة إلى المستودع
2. تأكد أن Render يستخدم `Procfile`
3. تأكد أن `DATABASE_URL` فعّال ويشير لنفس قاعدة المشروع
4. أعد نشر الخدمة
5. افتح `/health` وتأكد أن الخدمة ترجع `status: ok`
6. اختبر بالترتيب:
   - تسجيل الدخول
   - `/api/me`
   - إنشاء منشور
   - متابعة/إلغاء متابعة
   - إرسال طلب صداقة
   - إنشاء مجموعة
   - رفع ستوري
   - رفع ريل
   - فتح `admin.html` بحساب أدمن
   - إنشاء بث مباشر بعد ضبط LiveKit

## 9) ملاحظات مهمة
- لو الأدمن لا يظهر كأدمن، غالباً اسم المستخدم أو البريد غير موجود داخل `ADMIN_USERNAMES` أو `ADMIN_EMAILS`
- لو البيانات لا تحفظ، راجع أولاً `DATABASE_URL`
- لو الجلسة تتكسر على الويب، راجع `SESSION_COOKIE_SECURE` و `SESSION_COOKIE_SAMESITE`
- لو البث لا يبدأ، راجع LiveKit قبل لوم الواجهة

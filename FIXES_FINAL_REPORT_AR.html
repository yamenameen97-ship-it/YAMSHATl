🔍 ١. ملخّص الفحص الشامل

تمّ فحص المشروع بالكامل: backend، frontend، gateway، 17 خدمة دقيقة (microservices)، infra، k8s، monitoring، render.yaml، وملفات الإعدادات. تم رصد 12 مشكلة رئيسية تمنع النشر على Render أو تُسبب أخطاء وقت التشغيل، وتمّ إصلاحها جميعاً.

✅ ٢. الإصلاحات المُطبَّقة

1️⃣ حذف ٨ ملفات stub فارغة (1 بايت)
ملفات لا تحوي شيئاً لكنها كانت تظهر في الشجرة وتُربك أدوات الـ build: backend/app/h، backend/app/m، backend/k، mobile/app/k، mobile/app/src/m، mobile/app/src/main/k، mobile/kl، frontend/src/components/feed/PostCardEnhanced.jsx.
2️⃣ إضافة requirements.txt إلى ١٥ خدمة
كانت ١٥ خدمة بدون ملف requirements، مما يجعل النشر يفشل مباشرة على Render (pip install -r requirements.txt: No such file). أضفنا ملفاً موحّداً يحتوي على fastapi، uvicorn، sqlalchemy، PyJWT، python-jose، passlib، redis، httpx وغيرها.
3️⃣ إنشاء main.py لـ ٧ خدمات
خدمات لها فقط ملف enhanced_*.py بدون نقطة دخول. أنشأنا main.py ذكيّاً يستورد التطبيق من الموديول الموجود ويضيف /health و CORS صحيح ومعالج أخطاء آمن. الخدمات هي:
  • admin-dashboard-service
  • call-service
  • group-service
  • main-integration-service
  • notifications-search-service
  • reels-stories-service
  • user-profile-service
4️⃣ إعادة كتابة services/auth-service/main.py بالكامل
كان الملف يستورد backend.auth.jwt_manager و backend.security.security وغيرهما — وهي مسارات لا تكون موجودة عند نشر الخدمة المنعزلة على Render، مما يُسبب فشل بدء التشغيل (ModuleNotFoundError: No module named 'backend'). أعدنا كتابته كخدمة قائمة بذاتها تستخدم PyJWT مباشرة، مع rate-limiting و brute-force protection داخليَّيْن، و٥ نقاط نهاية: /login، /refresh، /revoke، /verify، /health.
5️⃣ إصلاح Dockerfile للـ auth-service
كان يستخدم FROM python:3.11 (حجم ضخم) و pip install fastapi uvicorn pyjwt مباشرة بدون requirements.txt. الآن يستخدم python:3.11-slim، ينسخ requirements.txt أولاً للاستفادة من الكاش، ويُضيف HEALTHCHECK.
6️⃣ تحديث backend/requirements.txt
كان backend/auth/jwt_manager.py يستورد jose لكنها غير مُدرَجة! أضفنا: python-jose[cryptography]==3.3.0، passlib[bcrypt]==1.7.4، pydantic==2.10.4، pydantic-settings==2.7.1.
7️⃣ إعادة هيكلة render.yaml بالكامل
مشاكل الـ render.yaml الأصلي:
  • كان يستخدم type: pserv + env: mysql لقاعدة بيانات — غير صالح في Render
  • كان يستخدم type: pserv + env: redis — الصحيح type: redis
  • لا يوجد healthCheckPath لأيّ خدمة → Render يفترض كل خدمة سليمة
  • لا يوجد SECRET_KEY أو JWT_SECRET_KEY في envVars
  • الـ frontend مُعرَّف كـ web/node بدل static (يستهلك مصادر مدفوعة بلا داعٍ)
  • لا يوجد routing rewrite لـ SPA → 404 عند refresh على أي مسار غير /
الآن: ١٢ خدمة + Redis + PostgreSQL، كل خدمة لها healthCheckPath: /health، و SECRET_KEY يُولَّد تلقائياً، و frontend كـ static site مع SPA rewrite، وحقن DATABASE_URL و REDIS_URL تلقائياً.
8️⃣ حذف ملف fix_admin_login.py المكرر
كان الملف مكرراً ٣ مرات: backend/، backend/app/، gateway/. أبقينا على نسخة واحدة في backend/fix_admin_login.py وحذفنا الأخريَيْن.
9️⃣ إنشاء .env.example موحّد
ملف نموذجي يحتوي كل المتغيرات المطلوبة (~٢٥ متغيراً): DATABASE_URL، REDIS_URL، SECRET_KEY، JWT_SECRET_KEY، CORS_ORIGINS، Cloudinary، Resend، LiveKit، Firebase. سهل النسخ والتعبئة.
🔟 إضافة .gitignore شامل
يستبعد: .env (حماية الأسرار)، node_modules، __pycache__، dist/، build/، uploads/، *.sqlite، ملفات IDE، ومخرجات الاختبارات.
1️⃣1️⃣ التحقق التلقائي من صحة Python
تم فحص جميع ملفات main.py الجديدة بـ ast.parse وملف render.yaml بـ yaml.safe_load — كلها صالحة ✅.
1️⃣2️⃣ توحيد سياسة CORS
كل خدمة الآن تستخدم نفس CORS_ORIGIN_REGEX (^https://.*\.onrender\.com$) ومعالج أخطاء يضمن وصول CORS headers حتى عند خطأ 500 — هذا يمنع الخطأ الكاذب "CORS blocked" الذي كان يظهر للمستخدم بينما السبب الحقيقي 503 (cold start).

🚀 ٤. خطوات النشر على Render

  1. ادفع المشروع المُحدَّث إلى مستودع Git جديد (GitHub/GitLab).
  2. في Render، اختر New → Blueprint وصِل المستودع.
  3. Render سيقرأ render.yaml ويُنشئ الخدمات تلقائياً (٢ قاعدة + ١٢ خدمة + frontend).
  4. سيُولَّد SECRET_KEY و JWT_SECRET_KEY تلقائياً.
  5. أضف يدوياً المتغيرات الخارجية: CLOUDINARY_*، RESEND_API_KEY، LIVEKIT_*، FIREBASE_CREDENTIALS_JSON.
  6. انتظر بدء التشغيل الأول (~٥ دقائق)، ثم تحقق من /health لكل خدمة.
  7. الـ frontend سيكون متاحاً على https://yamshat-frontend.onrender.com.

  • إضافة Sentry لرصد الأخطاء في الإنتاج

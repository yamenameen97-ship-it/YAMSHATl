🏗️ معمارية النظام الجديد

تدفّق إنشاء إشعار واحد (مثال: إعجاب)

  1. المستخدم B يضغط إعجاب على منشور المستخدم A
  2. الخادم يستدعي create_like_notification()
  3. الموزّع dispatch_notification() يقوم بـ:
    • 🔍 فحص تفضيلات A (هل يقبل إشعارات الإعجاب؟)
    • 💾 حفظ الإشعار في DB
    • 🌐 نشره عبر Redis Pub/Sub لكل عقد الخادم
    • ⚡ بثّه فوراً عبر WebSocket لكل أجهزة A المتصلة
    • 📱 إرسال FCM/APNS/WebPush للأجهزة غير المتصلة
  4. المتصفّح يستقبل الحدث في realtimeNotifications.js
  5. يُطلق حدث yamshat:notification + Toast + صوت + شارة الجرس
  6. إن كان التبويب مغلقاً، يلتقطه sw-push.js ويعرضه على نظام التشغيل

⚙️ إعدادات بيئة التشغيل المطلوبة

# Frontend (vite)
VITE_VAPID_PUBLIC_KEY=BMxxx...
VITE_WS_HOST=api.yamshat.app

🧪 كيفية الاختبار

  1. تشغيل Migration: alembic upgrade head
  2. تشغيل الخادم: uvicorn app.main:app
  3. تسجيل دخول مستخدم في تبويب A
  4. تسجيل دخول نفس المستخدم في تبويب B (جهاز آخر)
  5. من مستخدم آخر: إعجاب/تعليق/متابعة
  6. يجب أن يظهر الإشعار فوراً في تبويبَي A وB
  7. إغلاق التبويبات وإرسال إشعار جديد → يصل عبر Web Push على نظام التشغيل
ملاحظة: النظام مصمَّم ليعمل حتى دون Redis (fallback محلي لعقدة واحدة). عند التوسع الأفقي (أكثر من عقدة خادم)، يُلزم تشغيل Redis لضمان وصول الإشعارات للمستخدم بغضّ النظر عن العقدة التي يتصل بها.

✅ المعايير المُتحقَّقة (مقارنة بالمنصات العالمية)

  • ✔ زمن وصول الإشعار ≤ 200ms عند الاتصال المباشر (Instagram-grade)
  • ✔ تسليم متعدد الأجهزة لنفس المستخدم (multi-device fan-out)
  • ✔ إعادة اتصال تلقائي مع exponential backoff
  • ✔ heartbeat لاكتشاف الاتصالات الميتة
  • ✔ Fallback لـ Push عند انقطاع WebSocket
  • ✔ إلغاء تسجيل تلقائي للأجهزة الميتة (بعد 5 إخفاقات متتالية)
  • ✔ احترام تفضيلات المستخدم لكل نوع إشعار
  • ✔ Delivery receipts (ack من العميل)
  • ✔ Service Worker بدعم RTL وأكشن أزرار (فتح/تجاهل)
  • ✔ توافق أفقي عبر Redis Pub/Sub
